Introduction to PDML

Author: Christian Neumanns

Published: 2021-10-27

Introduction

The Practical Data and Markup Language (PDML) is a text format to store data.

PDML's design goals are:

  • human-friendly (easy to read and write for people)

  • suitable for

    • data and markup code

    • small and big, complex data structures

  • a basic format that is easy to parse/deserialize and serialize

  • unique, powerful extensions

A distinction is made between basic PDML and extensions. Basic PDML is the absolute minimum needed to store data. Extensions are optional features to make PDML more practical.

This document only covers basic PDML. However, chapter TODO contains a summary of extensions, and a link to more information.

Basic Examples

To get an idea of what can be done with PDML, let's have a look at some basic, simple examples.

Text Node

Suppose a config file in which parameter color has the value green.

In JSON we would use the following syntax:

"color" = "green"

In XML we could use an attribute:

color = "green"

... or an element:

<color>green</color>

In PDML the syntax is:

[color green]

The above code is called a node in PDML.

As we can see, a node is delimited by [] - a pair of square brackets. A node starts with [, and ends with ].

A node has a name and an optional value. In our example, the name is color and the value is green.

A space character is used to separate the name from the value.

Child Node

Besides text, a node's value can also be another node:

[config [color green]]

Here, the value of node config is another node with name color, and value green.

For better readability we can also write:

[config
    [color green]
]
Tree

The node's value can be a list of nodes, and each child node can itself have any number of child nodes:

[config
    [color green]
    [size
        [width 200]
        [height 100]
    ]
]

Hence, PDML can be used to store simple or complex tree data.

Text and Child Nodes Mixed

A node's value can be a mixture of any number of text and child notes. This makes PDML convenient to store markup code.

Consider the following HTML code:

<p>Life is <i>better</i> if we are <b>kind</b>.</p>

... which would be rendered as :

Life is better if we are kind.

In PDML this is written as:

[p Life is [i better] if we are [b kind].]
Empty Node

A node can be empty. It has a name, but no value:

[color]

In JSON this would be written as:

"color" = null

In XML:

<color></color>

or simply:

<color />

There is not much more to say about PDML's basic syntax.

For a formal and complete definition please refer to the TODO .

Versatility

Despite PDML's utmost simplicity, it is well suited to store many different kinds of data, such as:

  • configuration data

  • database tables

  • markup code

  • unstructured data

Examples are shown in TODO.

You can also have a look at the TODO source code of this article which is written in PML and uses the PDML syntax.

PDML can be converted to XML, and XML to PDML. Hence, all powerful XML technology can be applied. For example it's possible to read a PDML into an XML DOM and:

  • validate a document with XML Schema

  • query a document with XML Query

  • change the document (add, remove, and modify nodes) and write a modified version back to XML or PDML

  • transform a document with XSLT

Examples of how to do this are shown in the article TODO .

History of PDML

In 2018 I was a bit frustrated with existing markup languages (Markdown, Asciidoctor, HTML, Docbook, etc.) I used for my blog.

After some "pondering and using a pencil and eraser" sessions, I finally came up with a new markup language which I called Practical Markup Language (PML). In march 2019 I published We Need a New Document Markup Language - Here is Why. This article illustrates the problems I encountered with popular markup languages, and shows how they are solved in PML. The article also contained a link to a first implementation, so that people could try out PML and send feedback.

Later I realized that, besides markup code, the PML syntax could also be used to store all other kinds of structured and unstructured data. In March 2021 I published Suggestion For a Better XML/HTML Syntax (also published on codeproject). I named my idea practicalXML (pXML), because of the conceptual similarities between pXML and XML. Moreover, pXML could be converted to XML, and vice versa. All was published and documented at the (now obsolete) pXML website.

In October 2021 I decided to rename pXML to PDML. The reason was that pXML needed a lot of improvements (extensions) to make it suitable for PML (e.g. parameterized text, document splitting, raw text sections, etc). At the end, pXML was more than just an alternative syntax for XML. It had plugable and configurable types and extensions, as well as other features not available in XML. Thus the name was changed from practicalXML (pXML) to Practical Data and Markup Language (PDML), and everything was published on the TODO .

In a nutshell: PDML originated in PML, and was temporarily called pXML.
